# End-to-End TIA Portal → OpenCode → TIA Portal Integration Plan

## 1. Current State Assessment

### What Exists and Works
- **Contracts**: Complete DTOs, abstractions (`ITiaProjectService`, `IOpenCodeClient`, etc.), error codes, request types
- **Application**: `ContextService`, `CapabilityProvider`, `SessionManager`, `ObjectIdentityService`, `GuidIdGenerator`, `SystemClock`, `ContentHashService`
- **Simulator**: `SimulatorTiaProjectService` implementing `ITiaProjectService` + `ISelectionTokenFactory` with rich DemoConveyorLine data
- **MCP Tool Handlers**: `TiaContextTools`, `TiaReadTools`, `TiaReferenceTools`, `TiaCompileTools`, `TiaChangeTools` — all thin adapters to `ITiaProjectService`
- **McpHost**: ASP.NET Core app hosting MCP server with HTTP transport on port 43121
- **OpenCode Client**: `OpenCodeHttpClient` (HTTP) and `MockOpenCodeClient` (test double)
- **Add-In**: `ProjectTreeProvider` (behind `#if SIEMENS`, functional but blocks UI thread), `DevelopmentHost` (console dev host)
- **Tests**: Good coverage for Simulator, MCP tools, Application services, Contracts, Architecture

### Critical Issues Found

1. **MCP tools have no `[McpToolType]`/`[McpTool]` attributes** — OpenCode cannot discover them. The tool classes are plain C# classes registered via `WithTools<T>()` but lack the `ModelContextProtocol` attributes that expose them to MCP clients.

2. **No `tia_ping` diagnostic tool** — Cannot validate the MCP roundtrip independently.

3. **Add-In blocks the TIA Portal UI thread** — `ProjectTreeProvider.HandleAction` uses `GetAwaiter().GetResult()` on HTTP calls, which will freeze TIA Portal.

4. **No OpenCode orchestrator service** — The Add-In has inline HTTP calls to OpenCode with no abstraction, no health check, no task lifecycle management, no cancellation, no timeout.

5. **No application-level service for the OpenCode roundtrip** — `IOpenCodeClient` exists but has no implementation wired into the Add-In's command flow.

6. **No diagnostic commands** — No "Test OpenCode connection" or "Test MCP roundtrip" commands exist.

7. **No correlation ID propagation** — Correlation IDs are not threaded through the full flow.

8. **OpenCodeHttpClient assumes a REST API** — Endpoints like `/api/sessions`, `/api/sessions/{id}/tasks`, `/api/tasks/{id}/events` are assumed but not verified against OpenCode's actual API.

## 2. Architecture Decisions

### Decision 1: OpenCode as External Process
The Add-In connects to an already-running OpenCode server on `127.0.0.1`. It does NOT start OpenCode. The OpenCode server is configured separately (via `opencode.json`) to discover the MCP server at `http://127.0.0.1:43121/mcp`.

### Decision 2: Keep OpenCodeHttpClient Endpoints
The REST API pattern in `OpenCodeHttpClient` follows a reasonable convention for an agent runtime. We'll keep these endpoints and document them. If the real OpenCode API differs, the client can be swapped without changing the orchestrator.

### Decision 3: McpHost as Separate Process
The MCP server runs as a separate process (McpHost). The Add-In does NOT host the MCP server. This avoids net48/net8.0 framework conflicts and keeps the architecture clean.

### Decision 4: Simulator for Initial Validation
The initial validation uses `SimulatorTiaProjectService` in McpHost. When TIA Portal is available, swap to `TiaAgent.Openness.TiaProjectService` (same interface, different implementation).

### Decision 5: Async-First in Add-In
All OpenCode/MCP communication uses `async/await` with proper `SynchronizationContext` marshaling back to the UI thread. No `GetAwaiter().GetResult()`.

### T2: Create `tia_ping` Diagnostic MCP Tool

**New file:** `src/TiaAgent.Mcp/Tools/TiaDiagnosticTools.cs`

**Implementation:**
- `[McpToolType]` class `TiaDiagnosticTools`
- `[McpTool("tia_ping")]` method returning a `PingResult` with `status`, `source`, `tiaConnected`, `projectOpen`
- Delegates to `ITiaProjectService.GetCurrentContextAsync()` to check connectivity
- Returns structured JSON result
- Handles `TiaErrorException` gracefully (returns error status instead of throwing)

**Register in McpHost Program.cs and ServiceCollectionExtensions.cs**

### T3: Create OpenCode Integration Orchestrator Service

**New file:** `src/TiaAgent.Application/OpenCode/OpenCodeOrchestrator.cs`

**Interface:** `src/TiaAgent.Contracts/Abstractions/IOpenCodeOrchestrator.cs`

**Responsibilities:**
- Health check OpenCode before sending tasks
- Create sessions and start tasks via `IOpenCodeClient`
- Watch task events (SSE stream) and collect the final response
- Manage task lifecycle (started → progress → completed/failed)
- Timeout enforcement
- Cancellation support
- Correlation ID propagation
- Prevent duplicate active tasks

**Key methods:**
```csharp
Task<OpenCodeOrchestratorResult> ExecuteTaskAsync(OpenCodeTaskDescriptor descriptor, CancellationToken cancellationToken);
Task<bool> IsOpenCodeAvailableAsync(CancellationToken cancellationToken);
```

**DTOs:**
- `OpenCodeTaskDescriptor` — action name, message, selection token, metadata
- `OpenCodeOrchestratorResult` — success, response text, tool calls made, duration, error

### T5: Refactor ProjectTreeProvider for Async + Orchestrator

### T6: Implement "Explain Selected Object" Command

**Modify:** `src/TiaAgent.AddIn/Providers/ProjectTreeProvider.cs`

**Flow:**
1. Capture selected `IEngineeringObject` at click time
2. Generate immutable selection token
3. Build task descriptor with selection metadata (name, type, plcName)
4. Send to OpenCode orchestrator
5. OpenCode calls `tia_get_current_selection(selectionToken)` → gets snapshot
6. OpenCode calls `tia_read_block(objectId)` → gets block source
7. OpenCode generates explanation
8. Response displayed in Add-In UI

**Initial payload:**
```json
{
  "action": "explain_selected_object",
  "selectionToken": "sel-xxx",
  "selectedObject": {
    "id": "block-fb-conveyor",
    "name": "FB_Conveyor",
    "type": "FunctionBlock",
    "language": "SCL",
    "plcName": "PLC_1"
  }
}
```
