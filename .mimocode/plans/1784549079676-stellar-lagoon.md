# Plan: TIA Portal V21 Add-In — Functional Implementation — COMPLETED

**Status: COMPLETED (2026-07-20)** — Add-In verified working in TIA Portal V21. Context menus appear, "Test Integration" action executes successfully.

## Problem Analysis

The repository has a substantial codebase (76 C# files, 14 projects) but several critical issues prevent it from producing a working TIA Portal V21 Add-In:

### Root Causes of Failure

1. **Siemens assembly path mismatch** — `Directory.Build.props` defines `TiaPublicApiDir` pointing to `PublicAPI\V21\net48`, but `Siemens.Engineering.AddIn.Base.dll` lives in `PublicAPI\V21.AddIn\`. Both `TiaAgent.AddIn.csproj` and `TiaAgent.Openness.csproj` reference `AddIn.Base` from the wrong directory.

2. **No Config.xml manifest in source** — The `.addin` manifest is only generated at pack-time by `build.ps1`, which has bugs (string ID instead of GUID, missing transitive deps, wrong Publisher path).

3. **Broken packaging script** — `build.ps1 pack` tries the Siemens Publisher (often unavailable), falls back to plain ZIP (which TIA Portal won't load), and misses transitive NuGet DLLs.

4. **Redundant Placeholder.cs files** — Coexist with real implementations, causing potential type conflicts.

5. **Missing "Test Integration" action** — No simple, self-contained, verifiable action exists.

## Implementation Plan

### Step 1: Fix Assembly Reference Paths

**File: `Directory.Build.props`**
- Add `TiaAddInApiDir` property pointing to `$(ProgramFiles)\Siemens\Automation\Portal V21\PublicAPI\V21.AddIn`
- Keep existing `TiaPublicApiDir` for Openness assemblies (`V21\net48`)

**File: `src/TiaAgent.AddIn/TiaAgent.AddIn.csproj`**
- Change `Siemens.Engineering.AddIn.Base` reference to use `$(TiaAddInApiDir)` instead of `$(TiaPublicApiDir)`
- Keep `Siemens.Engineering.Base` reference from `$(TiaPublicApiDir)` (correct path)

**File: `src/TiaAgent.Openness/TiaAgent.Openness.csproj`**
- Change `Siemens.Engineering.AddIn.Base` reference to use `$(TiaAddInApiDir)`
- Keep `Siemens.Engineering.Base` and `Siemens.Engineering.Step7` from `$(TiaPublicApiDir)`

### Step 2: Remove Redundant Placeholder Files

**Delete:**
- `src/TiaAgent.AddIn/Placeholder.cs`
- `src/TiaAgent.Openness/Placeholder.cs`

These coexist with real implementations and could cause namespace conflicts.

### Step 3: Add "Test Integration" Action

**File: `src/TiaAgent.AddIn/Providers/ProjectTreeProvider.cs`**

Add a new `TiaAgentTestContextMenu` class (or extend existing `TiaAgentContextMenu`) with a "Test Integration" action that:
- Runs synchronously (no async, no external dependencies)
- Displays a MessageBox confirming the Add-In is loaded and functional
- Shows: Add-In name, version, timestamp, TIA Portal version (if available)
- Logs a diagnostic entry to `%LOCALAPPDATA%\TiaAgent\test_integration.log`
- Does NOT require any selected object (works from project tree root)

This action is self-contained — no MCP, no OpenCode, no external process needed.

### Step 4: Add Diagnostic Logging

**New file: `src/TiaAgent.AddIn/Diagnostics/AddInLogger.cs`**

Simple file-based logger that writes to `%LOCALAPPDATA%\TiaAgent\addin.log`:
- Add-In loaded/found events
- Menu action invocations
- Errors and exceptions
- Timestamps for all entries

This replaces the `NoOpLogger` in `AddInServices.cs` and provides visibility into what happens at load time.

**New file: `src/TiaAgent.AddIn/Config.xml`**

### Step 6: Rewrite Packaging Script

**File: `build.ps1` — `Invoke-Pack` function**

Replace the current broken packaging with a robust implementation:

1. Build the solution first (or verify it's built)
2. Collect ALL DLLs from `TiaAgent.AddIn\bin\Release\net48\` (includes transitive deps)
3. Collect dependency project DLLs from their respective `bin\Release\` dirs
4. Create OPC package using `System.IO.Packaging` (same approach as `BuildAddInManually.ps1`):
   - `/EngineeringVersion` → "21"
   - `/Product/Id`, `/Product/ProductName`
   - `/Permissions/Required/Tia/TIA.ReadOnly`
   - `/Meta/Version`, `/Meta/TimeStampUTC`, etc.
   - Feature assembly at root URI with full assembly name
   - Additional assemblies in `/LocalAssemblyCache/`
5. Sign with OpcSigner if certificate is available (optional — TIA Portal loads unsigned Add-Ins with a warning)
6. Copy to `%APPDATA%\Siemens\Automation\Portal V21\UserAddIns\`

### Step 7: Add Build Verification

**File: `build.ps1` — new `verify` command**

After packaging, verify:
- `.addin` file exists and is non-empty
- Config.xml is present inside the package
- Feature assembly (TiaAgent.AddIn.dll) is present
- All declared additional assemblies are present
- Package is valid OPC format

### Step 8: Update AddInServices Logging

**File: `src/TiaAgent.AddIn/AddInServices.cs`**

## External Blockers

- **Siemens assemblies required at build time** — If TIA Portal V21 is not installed, the net48 projects (`TiaAgent.AddIn`, `TiaAgent.Openness`) will fail to build because `Siemens.Engineering.Base.dll` etc. are not found. The build should gracefully handle this (skip Siemens-dependent projects) while still building the netstandard2.0 and net8.0 projects.
- **Siemens Publisher tool** — May not be available. The manual OPC packaging approach (`System.IO.Packaging`) bypasses this dependency.
- **OPC signing** — Optional. TIA Portal loads unsigned Add-Ins with a warning. The OpcSigner tool requires a certificate to be installed first.
