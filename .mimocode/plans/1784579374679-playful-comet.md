# Local Runtime Supervisor - Implementation Plan

## Executive Summary

Implement a PowerShell-based local runtime supervisor that orchestrates Bridge and OpenCode startup, manages ports, publishes a runtime manifest, and provides start/stop/status commands. The Add-In will be updated to discover services via the runtime manifest instead of hardcoded ports.

**Critical architectural fact:** Czarnak's `tia-mcp` runs via stdio (spawned by OpenCode as a child process), NOT as a separate HTTP service. The supervisor starts two processes (Bridge + OpenCode), not three. The MCP status is derived from OpenCode's health endpoint.

## Phase 1: PowerShell Supervisor Module

### 1.1 Module structure

Create `src/runtime/` directory:

```
src/runtime/
├── TiaAgent.Supervisor.psm1        # Main module (loads all functions)
├── TiaAgent.Supervisor.psd1        # Module manifest
├── Functions/
│   ├── Initialize-TiaAgentPaths.ps1    # Create directory structure
│   ├── Test-TiaAgentPrerequisites.ps1  # Validate environment
│   ├── Get-TiaAgentPort.ps1            # Port allocation with fallback
│   ├── New-TiaAgentRuntimeManifest.ps1 # Create/update runtime.json
│   ├── Start-TiaAgentService.ps1       # Start Bridge or OpenCode process
│   ├── Wait-TiaAgentHealth.ps1         # Health check with retries
│   ├── Read-TiaAgentSettings.ps1       # Load/validate settings.json
│   ├── New-TiaAgentOpenCodeConfig.ps1  # Generate opencode.generated.json
│   ├── Test-TiaAgentStaleRuntime.ps1   # Detect and clean stale state
│   ├── Lock-TiaAgentSupervisor.ps1     # Single-instance via named mutex
│   ├── Write-TiaAgentLog.ps1           # Structured logging
│   └── Stop-TiaAgentService.ps1        # Graceful process shutdown
└── Scripts/
    ├── run.ps1                         # Bootstrap + supervisor
    ├── stop.ps1                        # Controlled shutdown
    └── status.ps1                      # Status inspection
```

### 1.2 Key functions

#### `Initialize-TiaAgentPaths.ps1`
Creates the full directory structure under `%LOCALAPPDATA%\TiaAgent\`:
- `config/`, `runtime/`, `runtime/secrets/`, `logs/`, `scripts/`, `temp/`
- Copies scripts from source repo to `scripts/` if not present

#### `Test-TiaAgentPrerequisites.ps1`
Validates with actionable errors:
- Windows environment (`$IsWindows` or `$env:OS -eq 'Windows_NT'`)
- PowerShell >= 5.1
- `dotnet` CLI available and correct SDK (reads `global.json`)
- .NET Framework 4.8 Developer Pack installed (for Add-In)
- Bridge executable exists (`src/TiaAgent.Bridge/bin/Release/net8.0/TiaAgent.Bridge.dll`)
- `mimo` CLI available (`Get-Command mimo -ErrorAction SilentlyContinue`)
- `tia-mcp` CLI available (`Get-Command tia-mcp -ErrorAction SilentlyContinue`)
- `settings.json` exists and is valid JSON
- `config/opencode.json` exists and is valid
- `%LOCALAPPDATA%\TiaAgent` is writable

#### `Get-TiaAgentPort.ps1`
```powershell
function Get-TiaAgentPort {
    param(
        [string]$ServiceName,      # "bridge", "opencode"
        [int]$PreferredPort,       # e.g., 43119
        [int]$RangeStart = 43100,
        [int]$RangeEnd = 43200
    )
}
```
- Try preferred port first via `Test-NetConnection -ComputerName 127.0.0.1 -Port $port -InformationLevel Quiet`
- If unavailable, scan range 43100-43200 for free port
- Validate by attempting to bind a temporary `TcpListener`
- Return the allocated port

#### `New-TiaAgentRuntimeManifest.ps1`
- Atomic write: write to `runtime.json.tmp`, validate JSON, `Move-Item -Force`
- Schema version 1
- Updates only the fields the supervisor owns
- Never overwrites fields managed by other components

#### `Start-TiaAgentService.ps1`
```powershell
function Start-TiaAgentService {
    param(
        [string]$ServiceName,       # "bridge" or "opencode"
        [string]$ExecutablePath,
        [string[]]$Arguments,
        [string]$WorkingDirectory,
        [hashtable]$Environment,
        [string]$LogFile,
        [int]$StartupTimeoutSeconds
    )
}
```
- Starts process with redirected stdout/stderr to log files
- Captures PID
- Returns process object
- Sets `$LASTEXITCODE` on failure

#### `Wait-TiaAgentHealth.ps1`
```powershell
function Wait-TiaAgentHealth {
    param(
        [string]$Url,               # e.g., "http://127.0.0.1:43119/health"
        [int]$TimeoutSeconds = 30,
        [int]$RetryIntervalMs = 1000,
        [string]$ExpectedService    # "tia-agent-bridge" or "opencode"
    )
}
```
- Polls health endpoint with `Invoke-RestMethod -TimeoutSec 5`
- Bounded retries: `ceil(TimeoutSeconds / (RetryIntervalMs/1000))`
- Validates response contains expected service identity
- Returns `$true` on healthy, `$false` on timeout

#### `Lock-TiaAgentSupervisor.ps1`
- Uses .NET named mutex via `Add-Type`:
  ```csharp
  new System.Threading.Mutex(false, "Local\\TiaAgent.Supervisor", out bool created)
  ```
- Writes `supervisor.lock` with `instanceId`, `supervisorPid`, `startedAt`
- Returns mutex handle and instanceId
- On failure: checks if existing lock is stale (PID not running), cleans if stale

#### `Stop-TiaAgentService.ps1`
```powershell
function Stop-TiaAgentService {
    param(
        [System.Diagnostics.Process]$Process,
        [string]$ServiceName,
        [int]$GracefulTimeoutSeconds = 10,
        [string]$RuntimeInstanceId    # Verify process belongs to this instance
    )
}
```
- Validates PID still belongs to expected executable
- Calls `Process.CloseMainWindow()` for graceful shutdown
- Waits with bounded timeout
- Force-kills only if process doesn't stop and is owned by this runtime
- Updates runtime manifest

#### `Write-TiaAgentLog.ps1`
- Structured JSON log lines to `logs/supervisor.log`
- Fields: `timestamp`, `level`, `service`, `instanceId`, `correlationId`, `event`, `message`
- Never logs secrets

### 1.3 `run.ps1` - Bootstrap and Supervisor

```
1. Acquire supervisor mutex (Lock-TiaAgentSupervisor)
2. Create LocalAppData structure (Initialize-TiaAgentPaths)
3. Validate prerequisites (Test-TiaAgentPrerequisites)
4. Detect and clean stale runtime (Test-TiaAgentStaleRuntime)
5. Load settings (Read-TiaAgentSettings)
6. Allocate ports (Get-TiaAgentPort for bridge, opencode)
7. Generate transient credentials (MCP bearer token -> secrets/)
8. Publish runtime status as "starting" (New-TiaAgentRuntimeManifest)
9. Start Bridge (Start-TiaAgentService)
10. Wait for Bridge health (Wait-TiaAgentHealth on bridge /health)
11. Generate OpenCode config (New-TiaAgentOpenCodeConfig)
12. Start OpenCode (Start-TiaAgentService with generated config)
13. Wait for OpenCode health (Wait-TiaAgentHealth on opencode /health)
14. Publish runtime status as "ready"
15. Monitor child processes (loop checking health, update manifest on failure)
```

### 1.4 `stop.ps1` - Controlled Shutdown

```
1. Read and validate runtime.json
2. Confirm instanceId belongs to TiaAgent
3. Set global state to "stopping"
4. Stop OpenCode (graceful CloseMainWindow, bounded timeout)
5. Stop Bridge (graceful CloseMainWindow, bounded timeout)
6. Force-kill only owned processes that refused graceful stop
7. Remove transient secrets
8. Remove supervisor.lock
9. Remove or mark runtime.json as "stopped"
```

### 1.5 `status.ps1` - Status Inspection

```powershell
# Human-readable output (default):
TIA Agent Runtime

Instance   : d6b2893a
Status     : Ready
Supervisor : Running, PID 14820
Bridge     : Healthy, http://127.0.0.1:43119
OpenCode   : Healthy, http://127.0.0.1:43120

C:\Users\<user>\AppData\Local\TiaAgent\runtime\runtime.json

# JSON output (-Json flag):
{ "instanceId": "...", "status": "ready", "services": { ... } }
```

Validates each service: process exists, PID matches, executable matches, health endpoint responds.

### 3.3 Update `BridgeClientConfig.cs`

Add a fallback: if `addin.json` doesn't exist, try `RuntimeDiscovery.TryLoad()` to get Bridge URL and auth token from the runtime manifest + token file. This way the Add-In works both with and without the supervisor.

### 3.4 Update `AddInServices.cs`

When initializing `BridgeClientConfig`, also attempt runtime discovery:
1. Try loading `addin.json` (existing behavior)
2. If that fails, try `RuntimeDiscovery.TryLoad()` to get port/URL from manifest
3. Validate health endpoint before returning client
4. Show clear error if runtime unavailable

## Phase 4: Bridge Enhancements

### 4.1 Enhanced health endpoint

Update `BridgeController.HandleHealthAsync` to include `instanceId` and more detail:

```json
{
  "service": "tia-agent-bridge",
  "status": "healthy",
  "version": "0.1.0",
  "instanceId": "<from runtime manifest or generated>"
}
```

The Bridge reads the `instanceId` from environment variable `TIA_AGENT_INSTANCE_ID` set by the supervisor. This is a minimal change — the Bridge does NOT read `runtime.json` directly.

### 4.2 No other Bridge changes

The Bridge already supports configurable port via `bridge.json` and binds to loopback. The supervisor will set the port in `bridge.json` before starting Bridge, or pass it as a command-line argument if we add arg parsing. For MVP, the supervisor writes `bridge.json` with the allocated port before starting Bridge.

## Phase 5: Tests

### 5.1 New test project: `tests/TiaAgent.Runtime.Tests/`

Target: net8.0 (matching existing test projects)
Dependencies: xunit, FluentAssertions, Moq

Tests (no TIA Portal required):

1. **Port allocation tests** (`PortAllocatorTests.cs`):
   - Preferred port available -> returns preferred
   - Preferred port occupied -> returns next free
   - Entire range occupied -> throws
   - Port in range validation

2. **Manifest serialization tests** (`RuntimeManifestTests.cs`):
   - Roundtrip serialize/deserialize
   - Schema version validation
   - Status transitions valid
   - Missing required fields -> null/incomplete

3. **Atomic manifest write tests** (`ManifestWriterTests.cs`):
   - Writes temp file, validates, moves
   - Concurrent writes don't corrupt
   - Failed write leaves previous manifest intact

4. **Stale runtime detection** (`StaleRuntimeTests.cs`):
   - PID not running -> stale
   - PID running but wrong executable -> stale
   - Lock file older than timeout -> stale
   - Fresh lock with running PID -> not stale

5. **PID identity validation** (`ProcessIdentityTests.cs`):
   - Validate PID matches expected executable name
   - Validate PID belongs to expected instance

6. **Settings parsing tests** (`SettingsTests.cs`):
   - Default settings valid
   - Invalid port range -> error
   - Missing fields -> defaults
   - JSON parse error -> error

7. **OpenCode config generation tests** (`OpenCodeConfigTests.cs`):
   - Preserves agents, model, MCP config
   - Updates port correctly
   - Unrelated settings untouched

8. **Health retry tests** (`HealthCheckTests.cs`):
   - Immediate success -> returns true
   - Fails then succeeds -> returns true within timeout
   - Always fails -> returns false
   - Timeout respected

9. **Single-instance tests** (`SingleInstanceTests.cs`):
   - First lock succeeds
   - Second lock fails while first held
   - After release, new lock succeeds

10. **Shutdown tests** (`ShutdownTests.cs`):
    - Missing process -> graceful no-op
    - Stale PID -> skip kill
    - Process refuses graceful -> force kill after timeout

11. **Add-In runtime discovery tests** (`RuntimeDiscoveryTests.cs`):
    - Valid manifest -> parsed correctly
    - Invalid schema version -> null
    - Status not ready -> null
    - Missing file -> null
    - Health validation success/failure

### 5.2 Integration test

`SupervisorIntegrationTests.cs`:
- Starts lightweight fake processes (simple HTTP servers) on random ports
- Publishes health endpoints
- Runs the supervisor lifecycle (start, wait, verify, stop)
- Validates manifest state transitions

## Key Design Decisions

1. **PowerShell supervisor, C# tests** - PowerShell is the right tool for orchestration scripts on Windows; C# xunit is the right tool for testable logic
2. **No separate MCP Host process** - tia-mcp is stdio, spawned by OpenCode; MCP health is derived from OpenCode health
3. **Two services, not three** - Supervisor starts Bridge and OpenCode only
4. **Add-In stays net48 compatible** - RuntimeDiscovery uses manual JSON parsing (no System.Text.Json)
5. **Atomic manifest writes** - temp file + validate + move (PowerShell `Move-Item -Force` is atomic on same volume)
6. **Named mutex for single-instance** - .NET `System.Threading.Mutex` with `Local\` prefix
7. **Environment variable for instanceId** - Supervisor passes `TIA_AGENT_INSTANCE_ID` to Bridge, avoiding Bridge needing to read runtime.json
