# Plan: Integrate Czarnak/tia-portal-mcp as MCP Server

## Goal

Replace our custom MCP server (`TiaAgent.Mcp` + `TiaAgent.McpHost`) with [Czarnak/tia-portal-mcp](https://github.com/Czarnak/tia-portal-mcp) — a maintained, open-source MCP server for TIA Portal V21. This eliminates ~2,500 lines of custom MCP/Openness code while gaining batch operations, write safety tokens, and project lifecycle tools.

**Target flow:**
```
Add-In (user action) → HTTP → OpenCode/MiMoCode → stdio MCP → Czarnak's tia-mcp → TIA Portal
```

---

## What We Gain

- **~2,500 lines deleted** (Mcp, McpHost, Openness, Simulator projects)
- **Maintained OSS MCP server** (20 stars, MIT, NuGet package, CI)
- **Batch operations** (`execute_read_batch` — up to 50 reads per call, more efficient)
- **Built-in write safety** (preview-then-apply with `safetyToken`, audit trail)
- **Project lifecycle tools** (open, create, save, close, archive)
- **`tia-mcp doctor`** for environment validation
- **One fewer HTTP server** (stdio transport, no port 43121)

## Step 3: Rewrite Agent Profiles

The agent profiles must switch from our `tia_*` per-object tools to Czarnak's batch API.

### Tool Mapping

| Our Tool | Czarnak Equivalent | Via |
|---|---|---|
| `tia_get_current_context` | `get_project_status` | `execute_read_batch` |
| `tia_get_current_selection` | *None* — context embedded in prompt | — |
| `tia_list_blocks` | `browse_project_tree` | `execute_read_batch` |
| `tia_read_block` | `get_block_content` | `execute_read_batch` |
| `tia_get_block_interface` | `get_block_content` (includes interface) | `execute_read_batch` |
| `tia_get_call_hierarchy` | *No direct equivalent* | `read_cross_references` as partial substitute |
| `tia_find_references` | `read_cross_references` | `execute_read_batch` |
| `tia_compile_software` | `compile_check` | `execute_read_batch` |
| `tia_preview_block_change` | `preview_write_batch` | — |
| `tia_apply_approved_block_change` | `apply_write_batch` | — |
| `tia_ping` | `get_project_status` | `execute_read_batch` |

### Files to Rewrite

**`agents/tia-explain.md`** — Replace tool names with `execute_read_batch` + `get_project_status`. Add instructions for using `browse_project_tree` to discover block paths before calling `get_block_content`. Add note that object context comes from the prompt (no selection tokens).

**`agents/tia-review.md`** — Same as explain, plus `compile_check` access via `execute_read_batch`.

**`agents/tia-change.md`** — Same as explain, plus `preview_write_batch` / `apply_write_batch` for the change workflow. Czarnak's `safetyToken` replaces our `ApprovalToken`.

### Selection Token Gap

Our system captures selected objects via `selectionToken`. Czarnak's server has no concept of selection tokens — it works with explicit block paths (e.g., `PLC_1/Blocks/OB_Main`).

**Solution:** Embed object context directly in the agent prompt message:
```
The user selected object "OB_Main" of type "PlcBlock" in TIA Portal.
Please explain this object. Use browse_project_tree to locate it,
then read its content with get_block_content.
```

The `OpenCodeTaskDescriptor.SelectionToken` field becomes unused. The `Message` field carries the context instead.

## Step 4: Wire Add-In to Orchestrator (Fix Current Gap)

**File:** `src/TiaAgent.AddIn/Providers/ProjectTreeProvider.cs`

The current `HandleAction` writes a JSON file and shows a MessageBox with manual instructions. It should call the orchestrator directly.

**Change:** Replace the MessageBox + file write with:
```csharp
private async void HandleAction(string action, MenuSelectionProvider<IEngineeringObject> selection)
{
    var objects = selection.GetSelection();
    var enumerator = objects.GetEnumerator();
    if (!enumerator.MoveNext()) { /* show error */ return; }

    var selectedObj = enumerator.Current;
    var objName = selectedObj.ToString() ?? "Unknown";
    var objType = selectedObj.GetType().Name;

    var descriptor = new OpenCodeTaskDescriptor
    {
        Action = action,
        CorrelationId = $"tia-{Guid.NewGuid():N}",
        AgentId = action switch
        {
            "explain" => "tia-explain",
            "review" => "tia-review",
            "propose" => "tia-change",
            _ => "tia-explain"
        },
        Message = $"The user selected object \"{objName}\" of type \"{objType}\" in TIA Portal. " +
                  action switch
                  {
                      "explain" => "Please explain this object.",
                      "review" => "Please review this object for issues.",
                      "propose" => "Please propose improvements.",
                      _ => "Please analyze this object."
                  }
    };

    // Show progress UI, then call orchestrator
    var result = await AddInServices.Orchestrator.ExecuteTaskAsync(descriptor, CancellationToken.None);
    // Display result in UI
}
```

**Note:** TIA Portal menu callbacks must be fast (non-blocking). The orchestrator call should be fire-and-forget with a progress indicator, or run on a background thread with results displayed via a WPF window.

## Architecture After Integration

```
TIA Portal V21 (Add-In loaded)
  │
  │ User right-clicks → "Explain selected object"
  ▼
ProjectTreeProvider (net48, in TIA process)
  │ captures object name/type
  │ calls orchestrator
  ▼
OpenCodeOrchestrator
  │ HTTP POST to OpenCode
  ▼
OpenCode/MiMoCode (port 43120)
  │ spawns tia-mcp as child process
  ▼
Czarnak's tia-mcp (.NET 8, stdio)
  │ newline-delimited JSON IPC
  ▼
OpennessWorker (.NET 4.8)
  │ loads Siemens.Engineering.* DLLs
  ▼
TIA Portal Openness API
```

**Deleted from our repo:** TiaAgent.Mcp, TiaAgent.McpHost, TiaAgent.Openness, TiaAgent.Simulator, associated tests and scripts.

**Kept:** TiaAgent.AddIn, TiaAgent.Application (simplified), TiaAgent.OpenCode, TiaAgent.Contracts (simplified), agent profiles (rewritten).

## Open Questions

1. **MiMoCode MCP config format for stdio servers** — Need to verify the exact JSON schema. The current `config/opencode.json` may not be the native MiMoCode config. Check if MiMoCode reads from `~/.mimocode/settings.json` or similar.

2. **OpenCode HTTP API existence** — Our `OpenCodeHttpClient` assumes OpenCode exposes REST endpoints at port 43120 (`/api/sessions`, `/api/sessions/{id}/tasks`, `/api/tasks/{id}/events`). Verify this matches the actual MiMoCode API.

3. **TIA Portal menu callback threading** — Menu callbacks must be fast. The orchestrator call in `HandleAction` needs to be async/non-blocking with a progress indicator. May need a WPF window or background task pattern.

4. **Project path discovery** — How does the Add-In get the current `.ap21` project path to pass to the orchestrator? May need `TiaPortal.Projects.FirstOrDefault()?.Path`.
